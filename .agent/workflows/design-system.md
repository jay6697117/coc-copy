# /design-system — 系统设计引导

> 角色：game-designer（游戏设计师） + systems-designer（系统设计师）
> 来源：改造自 Claude Code Game Studios 的 `/design-system` Skill
> 参数：`/design-system [系统名称]`

## 前置条件
- `design/gdd/systems-index.md` 应该存在（运行 `/map-systems` 创建）
- 如无系统索引，会提示先创建

## 8 个必填章节

每个 GDD 必须包含以下 8 个章节，按顺序逐节编写：

| # | 章节 | 目的 |
|:--|:-----|:-----|
| 1 | **概述（Overview）** | 一段话说清这个系统是什么、为什么需要它 |
| 2 | **玩家幻想（Player Fantasy）** | 这个系统给玩家带来什么情感/体验？ |
| 3 | **详细规则（Detailed Rules）** | 完整的逻辑描述，不含歧义 |
| 4 | **公式（Formulas）** | 所有数值计算的数学定义，包含变量范围 |
| 5 | **边界情况（Edge Cases）** | 明确列出非标准状态的处理方式 |
| 6 | **依赖关系（Dependencies）** | 双向映射：依赖哪些系统 + 被哪些系统依赖 |
| 7 | **调试旋钮（Tuning Knobs）** | 暴露给策划的可调变量，方便平衡性调整 |
| 8 | **验收标准（Acceptance Criteria）** | 可测试的验证步骤，QA 能据此判断"做好了" |

## 逐节编写循环

对每个章节，执行以下循环（来自核心协作协议）：

```
上下文 → 提问 → 选项 → 决策 → 草稿 → 批准 → 写入
```

1. **上下文**：读取系统索引中该系统的依赖信息，读取已完成的相关 GDD
2. **提问**：针对当前章节提出 2-4 个澄清性问题
3. **选项**：为关键设计决策提供 2-3 个方案（含优劣分析）
4. **决策**：用户选择
5. **草稿**：编写该章节内容（3-5k token 以内）
6. **批准**：用户确认
7. **写入**：追加到 `design/gdd/[系统名].md`

### 特殊要求

- **公式章节**：
  - 所有公式必须明确定义每个变量和其取值范围
  - 给出至少 3 个示例值来演示公式效果
  - 标注哪些变量是"调试旋钮"

- **边界情况章节**：
  - 不允许模糊表述（如"系统应该感觉不错"）
  - 每个边界情况必须有明确的处理规则
  - 覆盖：零值、最大值、溢出、无效输入

- **依赖关系章节**：
  - 从系统索引中提取双向依赖
  - 标注依赖的具体接口（"需要角色系统提供攻击力数值"）

- **调试旋钮章节**：
  - 格式：变量名 | 默认值 | 范围 | 影响说明
  - 这些变量必须在代码中使用配置文件加载，不可硬编码
