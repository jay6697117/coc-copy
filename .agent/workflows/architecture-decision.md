# /architecture-decision — 架构决策记录

> 角色：technical-director（技术总监）
> 来源：改造自 Claude Code Game Studios 的 `/architecture-decision` Skill
> 参数：`/architecture-decision [决策主题]`

## 工作流步骤

### 步骤 1：定义决策上下文

询问或确认：
1. 需要做什么决策？（用一句话描述）
2. 为什么现在需要做这个决策？（触发因素）
3. 有哪些约束条件？

### 步骤 2：列出备选方案

提出 2-4 个可行方案，每个包含：
- 方案描述
- 优势
- 劣势
- 风险
- 对现有系统的影响

### 步骤 3：评估和推荐

从多个维度评估：
- 性能影响
- 开发成本
- 可维护性
- 可扩展性
- 与游戏支柱的一致性

给出推荐方案及理由。

### 步骤 4：记录决策

用户确认后，写入 `docs/architecture/ADR-NNN-[主题].md`：

```markdown
# ADR-NNN: [决策标题]

**状态**: [提议 | 已采纳 | 已废弃 | 已替代]
**日期**: [日期]
**决策者**: [角色]

## 上下文
[为什么需要这个决策？触发因素是什么？]

## 决策
[选择了什么方案？]

## 备选方案
### 方案 A: [名称]
- 优势：[...]
- 劣势：[...]

### 方案 B: [名称]
- 优势：[...]
- 劣势：[...]

## 后果
- 正面：[...]
- 负面：[...]
- 需要关注的风险：[...]

## 受影响的系统
- [列出受到影响的系统和文件]
```
