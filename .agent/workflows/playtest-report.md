# /playtest-report — 测试报告

> 角色：qa-lead（质量保证负责人）
> 参数：`/playtest-report [new|update]`

## 工作流步骤

### 步骤 1：收集测试信息

引导用户回答：
1. 测试的是哪个版本/构建？
2. 测试了哪些系统/关卡？
3. 测试时长？
4. 测试者是谁？（自己 / 外部测试者）

### 步骤 2：分类反馈

将反馈分为三大类：

#### 🐛 Bug（缺陷）
- 复现步骤
- 预期行为 vs 实际行为
- 严重程度：致命 / 严重 / 一般 / 轻微

#### 😤 UX 摩擦（用户体验问题）
- 哪个环节感觉不顺？
- 是否困惑？困惑在哪里？
- 建议改进方向

#### ⚖️ 平衡问题  
- 什么感觉太强/太弱？
- 什么感觉太快/太慢？
- 参考 GDD 中的调试旋钮

### 步骤 3：生成报告

写入 `production/releases/playtest-YYYY-MM-DD.md`：

```markdown
# 测试报告 — [日期]

## 基本信息
- 版本：[版本号/构建]
- 测试范围：[系统/关卡列表]
- 测试时长：[时间]
- 测试者：[名单]

## Bug 列表
| ID | 严重度 | 描述 | 复现步骤 | 状态 |
|:---|:-------|:-----|:---------|:-----|
| PT-001 | 严重 | [描述] | [步骤] | ⬜ 待修复 |

## UX 摩擦
| ID | 位置 | 描述 | 建议 |
|:---|:-----|:-----|:-----|
| UX-001 | [位置] | [描述] | [建议] |

## 平衡反馈
| ID | 系统 | 问题 | 相关旋钮 |
|:---|:-----|:-----|:---------|
| BAL-001 | [系统] | [描述] | [旋钮名] |

## 总体评价
[整体感受和最重要的改进方向]
```
