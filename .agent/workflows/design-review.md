# /design-review — 设计文档审查

> 角色：creative-director（创意总监）审查视角
> 来源：改造自 Claude Code Game Studios 的 `/design-review` Skill
> 参数：`/design-review [系统名称 或 文件路径]`

## 工作流步骤

### 步骤 1：读取目标文档

完整读取目标 GDD 文件（`design/gdd/[系统名].md`）。

### 步骤 2：读取项目上下文

- 读取 `contexts/context.md`（获取支柱和当前阶段）
- 读取 `design/gdd/systems-index.md`（获取依赖关系）
- 读取该系统的上游/下游 GDD（如存在）

### 步骤 3：8 维审查清单

对 GDD 的 8 个必填章节逐一进行审查：

| # | 章节 | 审查标准 |
|:--|:-----|:---------|
| 1 | **概述** | 是否一段话说清了系统目的？是否有歧义？ |
| 2 | **玩家幻想** | 情感描述是否具体？是否和游戏支柱一致？ |
| 3 | **详细规则** | 规则是否完整无遗漏？程序员能否仅凭此实现？ |
| 4 | **公式** | 变量是否有范围定义？是否有示例值？是否有除零风险？ |
| 5 | **边界情况** | 是否覆盖零值/最大值/无效输入？有无模糊笼统描述？ |
| 6 | **依赖关系** | 上下游是否和系统索引一致？接口是否明确？ |
| 7 | **调试旋钮** | 是否有默认值和范围？是否覆盖了关键平衡变量？ |
| 8 | **验收标准** | 是否可测试？QA 能否据此写测试用例？ |

### 步骤 4：额外检查

- **支柱一致性**：每个主要设计决策是否符合游戏核心支柱？
- **系统间一致性**：和已有 GDD 是否有矛盾或重叠？
- **可实现性**：在当前技术栈和团队规模下是否可行？
- **范围控制**：是否存在超出项目范围的功能蠕变？

### 步骤 5：发布审查报告

审查结果分为三个等级：

**✅ APPROVED — 通过**
> 所有章节完整、一致、可实现。可以进入实现阶段。

**⚠️ NEEDS REVISION — 需要修改**
> 大部分内容合格，但有 1-3 处需要调整。列出具体问题和修改建议。

**❌ MAJOR REVISION NEEDED — 需要大改**
> 存在结构性问题（缺少关键章节、和支柱矛盾、技术不可行等）。需要重新进入 `/design-system` 流程。

### 报告格式

```
📋 设计审查报告：[系统名称]

总体评估：[APPROVED / NEEDS REVISION / MAJOR REVISION]

✅ 通过项：
  - [通过的维度]

⚠️ 需要关注：
  - [具体问题] → [修改建议]

❌ 严重问题：
  - [具体问题] → [修改建议]

📊 支柱一致性：[通过/不通过]
📊 系统间一致性：[通过/不通过]
📊 可实现性：[通过/不通过]
```
