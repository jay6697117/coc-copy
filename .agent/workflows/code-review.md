# /code-review — 代码审查

> 角色：lead-programmer（首席程序员）
> 来源：改造自 Claude Code Game Studios 的 `/code-review` Skill
> 参数：`/code-review [文件路径 或 目录]`

## 路径对应的规范

代码审查会根据文件路径自动应用不同的编码规范：

| 文件路径 | 适用规范 |
|:---------|:---------|
| `src/gameplay/**` | `/standards-gameplay`（游戏逻辑规范）|
| `src/core/**` | `/standards-core`（核心引擎规范）|
| `src/ui/**` | `/standards-ui`（UI 规范）|
| `src/ai/**` | `/standards-ai`（AI/网络规范）|
| `src/networking/**` | `/standards-ai`（AI/网络规范）|
| `tests/**` | `/standards-test`（测试规范）|
| `prototypes/**` | 放宽审查标准（原型代码）|

## 审查维度

### 1. 架构合规性
- [ ] 是否符合技术架构文档（`docs/architecture/`）？
- [ ] 是否遵守关注点分离？（游戏逻辑不混入 UI、引擎核心不含业务逻辑）
- [ ] 依赖方向是否正确？（底层不应依赖高层）

### 2. 性能检查
- [ ] 热路径有无不必要的内存分配？
- [ ] 有无在每帧循环中做耗时操作？
- [ ] 大量对象有无使用对象池？
- [ ] 有无未释放的资源？

### 3. 数据驱动
- [ ] 游戏数值是否从配置文件/数据表加载？
- [ ] 有无硬编码的平衡数值？（damage = 50 ← 这是硬编码！）
- [ ] 调试旋钮是否已暴露？

### 4. 错误处理
- [ ] 空值/边界情况是否处理？
- [ ] 有无未捕获的异常？
- [ ] 错误消息是否有用？

### 5. 代码风格
- [ ] 命名约定是否一致？
- [ ] 注释是否使用中文？
- [ ] TODO/FIXME 是否有署名？（格式：`TODO(name): 描述`）
- [ ] 函数长度是否合理？（推荐 < 30 行）

### 6. 与 GDD 一致性
- [ ] 实现是否和对应的 GDD 一致？
- [ ] 有无偏离设计文档的地方？（如有，是否记录了 ADR？）

## 输出格式

```
🔍 代码审查报告

📁 审查范围：[文件/目录]
📏 适用规范：[standards-xxx]

✅ 通过项（[N] 项）
  - [通过的维度]

⚠️ 建议改进（[N] 项）
  - [文件:行号] [问题描述] → [建议]

❌ 必须修复（[N] 项）
  - [文件:行号] [问题描述] → [修复方案]

总评：[通过 / 需修改 / 重写]
```
